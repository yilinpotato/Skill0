# SkillRL ALFWorld 技能内化项目架构说明

## 1. 项目范围

本项目当前面向 ALFWorld text-only 环境中的小模型技能内化训练与评测，目标是在不依赖外部检索服务的前提下，将 `memory_data/alfworld/claude_style_skills.json` 中的技能知识逐步转移到策略模型参数中。

当前工作重点包括：

- 基于 `SkillRL` + `GRPO` 的 ALFWorld 单阶段强化学习训练。
- 基于 `skills_only_memory` 的技能注入、逐步撤除与在线帮助性评估。
- 面向 Qwen3 Thinking 模型的 prompt、action projection、日志与可观测性适配。
- 训练态、冒烟态与纯 API 评测态三套可独立运行的流程。

当前主训练模型默认配置为：

```text
/GLOBALFS/hit_wxia_1/.cache/modelscope/hub/models/Qwen/Qwen3-4B-Thinking-2507
```

当前主训练脚本：

```text
examples/grpo_trainer/run_alfworld_fixed_single_stage.sh
```

默认输出目录：

```text
/GLOBALFS/hit_wxia_1/myl/SkillRL/skillrl_outputs/skillrl_mvp/<EXPERIMENT_NAME>
```

## 2. 总体架构

训练主链路如下：

```text
ALFWorld Text Environment
        |
        v
SkillRL Environment Manager
        |
        v
skills_only_memory 检索与 prompt 注入
        |
        v
Qwen3-4B-Thinking-2507 + LoRA Actor
        |
        v
vLLM Rollout + 多轮环境交互
        |
        v
GRPO 更新 / Validation / W&B / 本地日志
```

关键模块职责如下：

- `agent_system/environments/env_package/alfworld/`
  - 负责 ALFWorld 环境封装、admissible action 获取、projection 与 reward 相关对接。
- `agent_system/memory/skills_only_memory.py`
  - 负责技能库加载、global skill 注入、task-specific skill 注入、技能数量控制与 active id 过滤。
- `agent_system/multi_turn_rollout/rollout_loop.py`
  - 负责每步 prompt 构造、模型输出解析、trajectory/context 记录、W&B 表格日志。
- `verl/trainer/ppo/ray_trainer.py`
  - 负责训练循环、validation、skill internalization、在线帮助性评估、metrics 汇总与记录。
- `verl/utils/tracking.py`
  - 负责 W&B 初始化、timeout 配置与统一 logging 接口。

## 4. 当前训练流程

### 4.1 主训练脚本

推荐启动方式：

```bash
cd /GLOBALFS/hit_wxia_1/myl/SkillRL
source $HOME/miniconda3/etc/profile.d/conda.sh
conda activate skillRL

CUDA_VISIBLE_DEVICES=0 \
bash examples/grpo_trainer/run_alfworld_fixed_single_stage.sh
```

主训练脚本执行顺序如下：

1. 设置模型、训练、W&B 与资源参数。
2. 执行 `scripts/setup_alfworld_cache.sh` 准备项目内 ALFWorld cache。
3. 创建运行目录、W&B 目录与 diagnostics 目录。
4. 可选启动 `scripts/resource_monitor.sh` 做资源采样。
5. 调用 `examples.data_preprocess.prepare` 生成 parquet 数据。
6. 构造 Hydra 覆盖参数并启动 `verl.trainer.main_ppo`。
7. 在训练过程中周期性执行 validation、W&B 记录、本地日志记录与 checkpoint 保存。

## 6. Prompt、上下文与技能注入

### 6.1 Prompt 模板结构

ALFWorld prompt 模板位于：

```text
agent_system/environments/prompts/alfworld.py
```

当前有三类模板：

- `ALFWORLD_TEMPLATE_NO_HIS`
  - 首步使用，无历史。
- `ALFWORLD_TEMPLATE`
  - 有历史但不带 memory。
- `ALFWORLD_TEMPLATE_WITH_MEMORY`
  - 有历史并带技能 memory。

当前针对 Qwen3 Thinking 模型的关键约束已写入模板：

- 响应可能已经处于打开的 `<think>` 块内。
- reasoning 必须非常短。
- 应立即输出 `</think><action>exact admissible command</action>`。
- 不允许在 `<action>` 中输出解释性文本。

### 6.2 上下文内容结构

每一步模型实际看到的核心结构为：

1. 任务描述。
2. 检索到的技能文本。
3. 最近 `history_length` 步的 observation/action 历史。
4. 当前 observation。
5. 当前 admissible actions。
6. 输出格式约束。

首步通常不带 history，且默认不带 memory；后续步骤在启用 `skills_only_memory` 后会进入 `ALFWORLD_TEMPLATE_WITH_MEMORY` 路径。

### 6.3 技能注入机制

技能库文件：

```text
memory_data/alfworld/claude_style_skills.json
```

包含三部分：

- `general_skills`
- `task_specific_skills`
- `common_mistakes`

`skills_only_memory.py` 当前支持：

- global skill 数量上限 `global_skill_top_k`
- global skill 显式 active id 过滤
- task-specific skill 显式 active id 过滤
- 模板模式与 embedding 模式两种检索方式

当前主训练脚本默认使用：

- `top_k=12`
- `task_specific_top_k=null`
- `global_top_k_schedule=[12,12,12,8,8,8,6,6,6,4,4,4,2,2,2,0]`

这意味着：

- global skills 会按 schedule 逐步减少。
- task-specific skills 默认不会因 `top_k` schedule 自动减少。

## 7. Action Projection 与 Qwen3 Thinking 适配

Action projection 位于：

```text
agent_system/environments/env_package/alfworld/projection.py
```

当前已做的适配包括：

- 支持从 `<action>...</action>` 中提取动作。
- 若模型未输出 `<action>` 标签，则回退到全文匹配 admissible actions。
- 支持从冗长表达中恢复标准动作，如 `I will go to drawer 1 now`。
- 若成功恢复到 admissible action，则允许继续环境交互。
- 若输出中包含中文字符，则仍标记为 invalid。

该机制用于缓解 Qwen3 Thinking 模型 reasoning 偏长、动作标签缺失或动作文本被包裹在自然语言中的问题。

## 9. 技能内化机制

### 9.1 Schedule 内化

当前原始的“渐进式技能内化”是基于 global skill 数量调度实现的。

机制：

- 训练过程中按 `global_top_k_schedule` 修改可见 global skills 的数量。
- 当 `skillzero_include_task_specific=True` 时，该机制会同时把当前 task type 的 task-specific skills 也纳入“带技能 / 去技能”对比评估。

示例：

```text
[12,12,12,8,8,8,6,6,6,4,4,4,2,2,2,0]
```

表示随着训练推进，global skills 从 12 条逐步降到 0 条。

### 9.2 SkillZero 风格对比评估

当 `GLOBAL_INTERNALIZATION_MODE=skillzero` 时，validation 额外执行一组 `skills_off` 对比评估：

- `val`
  - 当前配置下的正常验证结果。
- `val/skillzero/skills_off/...`
  - 强制关闭当前注入的 external skills 后的验证结果。
- `val/skillzero/skills_on/...`
  - 正常注入 external skills 时的验证结果。
- `val/skillzero/delta/...`
  - 两者差值。

当前默认 `skillzero_include_task_specific=True`，因此这里的 external skills 包括：

- global skills
- 当前 `eval_task_type` 对应的 task-specific skills

对应训练脚本环境变量：

```text
SKILLZERO_INCLUDE_TASK_SPECIFIC=True
```

该机制用于观察“关闭当前所有外部技能注入后模型还剩多少能力”。

### 9.3 在线帮助性评估

当前已实现的 on-policy helpfulness evaluation 只覆盖：

- `general_skills`

实现位置：

```text
verl/trainer/ppo/ray_trainer.py
agent_system/memory/skills_only_memory.py
```

评估逻辑：

1. 先执行一次正常 validation，得到 base metric。
2. 对候选技能逐条做“去掉该技能后”的额外 validation。
3. 计算：
   - `with_skill`
   - `without_skill`
   - `delta = with_skill - without_skill`
4. 若 `delta` 足够小，则认为该技能当前帮助有限，streak 增加。
5. 若开启自动内化，且连续若干轮都满足阈值，则从 active skills 中移除。

### 9.4 在线帮助性评估的触发时机

帮助性评估不是每个 rollout 都执行，而是跟随 validation 周期执行。

当前触发条件：

- `trainer.test_freq > 0`
- 且满足以下之一：
  - `global_steps % trainer.test_freq == 0`
  - 到达最后一个训练 step

当前主脚本默认：

```text
TEST_FREQ=25
```

因此默认行为是：

- 每训练 `25` 个 step 评估一次技能帮助性。
- 最后一个 step 再额外评估一次。

另外：

- `trainer.val_before_train=True`
- 但在线帮助性评估挂接在训练循环中的 validation 分支，因此主要按训练中 validation 周期触发。

### 9.5 每轮评估多少技能

当前实现固定为：

- 每轮 validation 全量评估当前所有 active global skills
- `task-specific skills` 不参与 `on-policy helpfulness`，但会参与 `skillzero` 的 `skills_on / skills_off` 学习对比

这样可以在同一轮 validation 中直接比较所有 global skills 的 `delta`，判断哪一条 skill 被学到最多、哪一条仍然最有帮助。

### 9.6 自动内化相关参数

当前支持的关键参数如下：

```text
ONPOLICY_HELPFULNESS_EVAL_ENABLED
ONPOLICY_HELPFULNESS_AUTO_INTERNALIZE
ONPOLICY_HELPFULNESS_DELTA_THRESHOLD
ONPOLICY_HELPFULNESS_PATIENCE
ONPOLICY_HELPFULNESS_REMOVE_PER_ROUND
ONPOLICY_HELPFULNESS_MIN_ACTIVE_SKILLS
```

语义说明：

- `DELTA_THRESHOLD`
  - 去掉技能后性能下降不超过该阈值时，认为该技能“帮助有限”。
- `PATIENCE`
  - 连续多少个评估轮次都满足阈值后，才允许移除。
- `REMOVE_PER_ROUND`
  - 每轮最多移除多少条 global skill。
- `MIN_ACTIVE_*`
  - 保底保留的最小 active 技能数量。

### 9.7 性能成本

在线帮助性评估会增加 validation 成本。

若一轮正常 validation 成本记为 `1x`，则一轮完整帮助性评估成本近似为：

```text
1 + 当前 active global skill 条数
```

例如当前 active global skills 为 `12` 条，则一次 validation 约为：

```text
13x
```

因此该机制适合在以下场景下使用：

- `TEST_FREQ` 设得较低
- global active skill 数量已经下降
- 需要精确比较全部 global skills 的学习程度

## 11. 模型输出、valid action 与 done 处理

### 11.1 `is_action_valid` 的含义

`is_action_valid` 表示 projection 之后该步动作是否被判定为合法动作。

当前实现中：

- 若成功匹配到 admissible action，通常会记为 `1`。
- 若存在中文字符等异常形式，则仍可能记为 `0`。
- `env_action` 是环境真正执行的动作，可能与 `model_output` 不同。

因此出现“动作执行成功但 `is_action_valid=0`”通常意味着：

- 原始输出格式不符合规范。
- 但 projection 从文本中恢复出了可执行动作。

### 11.2 `done` 与轨迹停止

当前 rollout 已修复为：

- 一旦环境返回 `done=True`，后续不会继续在该 trajectory 上累积伪步骤。
- `active=false` 的步仅作为结果展示，不应再参与后续有效决策统计。

### 11.3 为什么 Thinking 模型容易出现长输出

Qwen3 Thinking 模型的聊天模板会默认进入 `<think>` 状态，且模型倾向于生成较长 reasoning。因此当前已做两层约束：

1. Prompt 层面要求 reasoning 极短并立即闭合 `</think>`。
2. Projection 层面对漏标签或冗长输出做动作恢复。

## 12. 训练产物与后续使用

训练输出位于运行目录下的 checkpoint 子目录，例如：

```text
/GLOBALFS/hit_wxia_1/myl/SkillRL/skillrl_outputs/skillrl_mvp/<EXPERIMENT_NAME>/global_step_<N>/
```

常见使用方式：

- 继续训练：复用原 `EXPERIMENT_NAME`。
- 离线评测：将 checkpoint/LoRA 作为模型权重供推理服务加载后，配合 `scripts/eval_alfworld_api.py` 测试。
- 分析训练行为：结合 `metrics.jsonl`、`trajectories.json`、`contexts.json` 与 W&B 表格查看。

## 13. 已知限制

当前仍需明确的限制如下：

- 当前“线性衰减预算”实现的是 global skill 数量调度，不是按 token 数做硬预算裁剪。
- task-specific skills 的 schedule 衰减未单独实现，当前不会参与 `on-policy helpfulness`，但会参与 `skillzero` 对比评估。
- `normalize_reward` 与 `reward_clip` 当前主要作为配置记录保留，训练代码尚未完整消费这两个字段。
- Qwen3 Thinking 模型虽已做 prompt 与 projection 适配，但仍可能出现 reasoning 过长、action 标签缺失或格式偏移，需要结合 `MAX_RESPONSE_LENGTH` 与日志持续观察。
